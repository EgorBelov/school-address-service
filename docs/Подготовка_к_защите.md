## 3. На что делать упор (сильные стороны)

1. **Принцип «структура до LLM» — главное архитектурное решение.** Типовые таблицы (Пермь) разбираются детерминированно, бесплатно и мгновенно; LLM включается только там, где иначе не формализовать (Балашиха, Берёзники). Это и экономия, и предсказуемость, и отличие от чисто-LLM-решений.
2. **Полный цикл и проверяемость.** От импорта произвольного постановления (.doc/.docx/.pdf, в т.ч. скан) до поиска школы с **правилом-основанием** и ссылкой на документ. `house_rule_raw` хранится всегда → результат можно сверить и поправить.
3. **Устойчивость на всех стыках.** DaData → локальный regex; GigaChat → retry 4× с backoff; скан-PDF → OCR; ошибка одного чанка не валит документ (мягкая деградация); автоматический разбор **не пишется в БД напрямую** — проверка оператором.
4. **Реальные данные, а не игрушечные.** 3 муниципалитета, 3 формата, 2 структурные стратегии + LLM, ≈700 правил, 4 реальных постановления.
5. **Зрелый процесс.** WBS/Ганта/RACI/реестр рисков/DoD, code review перед merge, пирамида тестов (34 кейса, 5 видов), журнал из 10 дефектов.
6. **Производительность с запасом.** Медиана ≈50 мс, P95 ≈180 мс при цели NF-01 < 1 с (запас ~5×).

## 4. Слабые места — снять проактивно

Лучше **самим назвать** в заключении как «осознанный технический долг учебного прототипа + бэклог», чем ждать вопроса:

- **Нет количественной оценки качества LLM** (precision/recall на размеченном эталоне) — оценка качественная на 4 постановлениях. → План: разметить выборку, добавить few-shot.
- **Нет аутентификации админ-панели** (BUG, принят как ТД). → В бэклоге первым пунктом.
- **Строковое хранение даты постановления** (ТД). → Alembic + перевод в DateTime.
- **SQLite + линейный обход правил** в памяти. → PostgreSQL + SQL-индекс по нормализованной улице.
- **OCR/цепочки LLM синхронны** — долгая загрузка. → Перевод в фоновые задачи.

## 5. Потенциальные вопросы и ответы

### Архитектура и выбор LLM
**Q-A1. Почему GigaChat, а не GPT-4 / Claude / YandexGPT?**
Официальная доступность из РФ без обходных решений; бесплатный лимит покрывает учебный проект; достойное качество на русскоязычной юриспруденции. Задача — извлечение структуры из текста (information extraction), а не творчество, поэтому топовая модель не требуется.

**Q-A2. Почему не отдать LLM весь разбор, включая Пермь?**
Многоколоночные таблицы Перми разбираются структурно — детерминированно, бесплатно, мгновенно и без риска галлюцинаций. LLM — это деньги, задержка и недетерминизм. Принцип: структура → и только если не разобрать → LLM.

**Q-A3. Почему запрос на каждую школу, а не один по всему документу?**
Компактный запрос на школу/чанк надёжнее (меньше путаницы, меньше пропусков), дешевле по токенам и даёт мягкую деградацию — сбой одного чанка не теряет весь документ.

### Качество и доверие к LLM
**Q-К1. Как гарантируете, что LLM не «придумает» правило?**
Три рубежа: (1) промпт требует строго JSON фиксированной схемы; (2) `clean_json_text` + `json.loads` + **валидация Pydantic v2** — недопустимые значения отсекаются на границе с БД; (3) `house_rule_raw` сохраняется всегда, и разбор проходит **обязательную проверку оператором** через `/admin/parse-with-ai` перед сохранением.

**Q-К2. Какова точность распознавания? Есть метрики precision/recall?**
Честно: формального количественного бенчмарка в текущей версии нет — оценка качественная на 4 реальных постановлениях трёх форматов. Наблюдаемые типичные ошибки: пропуск улицы из длинного перечня, склейка двух улиц, неточная чётность. В развитии — количественная оценка на размеченной выборке и few-shot-примеры сложных случаев. *(Не выдавать цифру, которой нет.)*

### Надёжность
**Q-Н1. Что происходит при недоступности GigaChat?**
Transient-ошибки (SSL EOF, 429, 5xx, обрывы) повторяются: до 4 попыток с экспоненциальной задержкой 4/8/16/32 с (+2,5 с anti-throttle между запросами). Логические ошибки (битый JSON) не ретраятся. Если разбор всё же не удался — школы из проблемного чанка остаются оператору, остальные сохраняются.

**Q-Н2. Что при недоступности DaData?**
Поиск переключается на встроенный regex-парсер адреса — пользователь сбоя не замечает (NF-02).

**Q-Н3. Что значит «три уровня fallback»?**
Это три независимых внешних сервиса, у каждого — деградация: DaData→локальный regex; GigaChat→retry, при отказе разбор остаётся оператору; ocr.space используется только для скан-PDF без текстового слоя. Ядро (поиск по уже загруженным правилам) работает полностью офлайн.

### Безопасность и эксплуатация
**Q-Б1. Безопасность? Любой может зайти в админку?**
Path traversal при загрузке файла закрыт (BUG-001/002). Отсутствие аутентификации админ-панели — известный технический долг учебного прототипа, первый пункт бэклога; в продакшене закрывается авторизацией.

**Q-Б2. Масштабируемость? SQLite потянет регион?**
Сейчас 700 правил, P95 ≈180 мс — запас ~5× к NF-01. Для многопользовательской эксплуатации в бэклоге: переход на PostgreSQL, SQL-индекс по нормализованной улице (вместо линейного обхода), фоновые задачи для OCR/LLM.

### Данные, UX, отличие от аналогов
**Q-Д1. Чем лучше сервиса «Найди свою школу» на Госуслугах?**
Открытый доступ для специалистов УО, импорт **произвольных** постановлений (не закрытый алгоритм), выдача **правила-основания** для проверяемости, потенциальный открытый API. Госуслуги/ГИС-порталы не закрывают цикл «импорт документа → поиск».

**Q-Д2. Зачем «толерантный матч локалити», если есть DaData?**
DaData часто ошибается на падежах и «ё» в названиях населённых пунктов. Авторская техника сопоставляет «Пермь↔Перми», «Берёзники↔Березников», «Балашиха↔Балашихи» — иначе адрес из DaData не сматчился бы с локалити правила.

**Q-Д3. Повторная загрузка постановления не задвоит правила?**
Нет, загрузка идемпотентна — повтор не создаёт дублей.

**Q-Д4. Как извлекаете данные из скан-PDF без текстового слоя?**
Если `pdfplumber` вернул пустой текст — OCR через ocr.space, затем тот же конвейер.
